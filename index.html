学习

  1. 开始看《你不知道的JS中卷》
    第一部分价值不大，就简单的看了看。第二部分异步和性能讲的很精彩，目前在二刷，如要完全理解估计还要刷几遍。看了这本书才知道，promise原来有这么大的意义，它的出现才意味着JS有了自己的异步。

  2. 了解Immutable
    immutable是配合react使用的一个库，拥有自己的数据结构，可以让state变得更加易于追踪，并且减少不必要的渲染从而提高程序的性能。我会来了解immutable，是因为在项目中遇到了由于immutable使用不当而导致的bug。查阅了一些资料后，现在对它的常用方法和使用场景都有了一定的了解。我对immutable的理解是，它实现了一些数据结构，数据结构内部的存取算法可以最大程度的节省开销，只变化变了的位置。这对于react的意义就是，精确识别到state的变化位置，这样只会触发这部分变化相关的重新渲染。除此之外，每次存取immutable都会返回一个新的对象，这样就便于追踪state的变化历程了。

详解vue-从熟悉到熟练-part2-对象的选项们

发表于 2018-04-09

前文我们知道，在vue单文件组件中，最终会导出一个对象。

vue已经为我们预先指定好了这个对象一些选项，如data, components, methods, computed, watch, mixins, filters, 钩子函数等等。本文我们将从组件的使用场景出发，详解这些属性和方法的用法。

组件内部属性

这里的内部指的是，组件自用的，不与外部产生交互的属性。如果我们想构造出一个不与外部交互的静态页面，那么只使用这里的选项就可以了。

data用于保存组件内部状态

对于一个组件，它独有的内部状态全部保存在data中。组件被创建后，你可以在任何地方更改data中的状态。与状态绑定的一切都会跟随状态的变化响应式地变化：绑定的DOM会重新渲染，绑定的计算属性(computed)会自动更新，对应的观察者（watch）会自动执行。

//logTimeInterval.js
import moment from ‘moment’;
import { LOG_INTERVAL } from ‘@/constants/application’;

export default {
data() {
return {
LOG_INTERVAL,
interval: LOG_INTERVAL.ONEWEEK.type,
};
},
computed: {
targetTimeInterval() {
const { time, unit } = LOG_INTERVAL[this.interval];
return {
from: moment().subtract(time, unit).valueOf(),
to: moment().valueOf(),
};
},
},
};

//组件

export default {
mixins: [logTimeInterval],
data() {
return {
content: ‘timeInterval:’,
};
},
};

//组件

export default {

学习

大致看了下《clean code》。书是以Java的视角讲代码整洁的，我只能看明白大致的思路。我的理解是，整洁的代码一定是读起来像文档一样的。实践中关注比较多的是变量/函数命名和一个函数只做一件事情。好的命名可以让人只看名字不看细节就知道变量/函数是做什么的；一个函数只做一件事情可以将复杂的逻辑分离，让代码更易读易测易扩展。
举个简单栗子。

背景：一个确定按钮，它的点击事件可能是创建新的规则转发操作，也可能是更新一个已有规则转发操作。

实践：

首先从HTML上我们就可以得知，按钮的click事件会触发规则转发。
然后看toggleRuleForward这个函数做了什么：

  1. 构造请求所需的参数

  2. 根据转发类型（是否为创建）判断调用哪个API。
    这里将构造请求参数，判断操作类型以及调用API的逻辑抽出函数之外，保持toggleRuleForward内部逻辑清晰。他的任务就是负责判断当前应该调用哪个API，真正的收发请求是在函数createRuleForward和updateRuleForward中处理的。对于这样的toggleRuleForward，相信不需要去看getForwardData，isCreateForward，createRuleForward，updateRuleForward中的实现细节你也会知道他们做了什么。

    再来说说敏捷实践中非常重要的一环：拆tasking。刚加入骚窝的时候我并不太懂拆tasking的意义，并且意识也不够，经常略过这个环节直接开搞，给自己造成了很多麻烦。后来通过自己亲手挖的坑，才理解了拆tasking的意义。然后发现，其实它和clean code也关系密切。

    所谓拆tasking，大白话就是在明确需求后，开始实现前，将任务拆解成一个个的小单元，不考虑细节，只明确每个单元的输入输出。然后单元之间通过输出输出彼此连接起来，就完成了最终的任务。所以其实拆tasking就是在抽离逻辑，一个单元就是一个“只做一件事情”的函数。同时它也可以帮我们整理思路，遇到问题定位起来也更容易。
